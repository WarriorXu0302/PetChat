# Xinxin v2.1 失败修复 Playbook

适用：`qa/v2.1-qa-checklist.md` 任一检查项未通过时，按严重度分批处理。

## A. 身份漂移类

### 症状
- 发型、脸型、服装比例、配件位置在不同帧有明显差异
- 某一状态看起来像不同角色版本

### 修复顺序
1. 先暂停所有非 base 作业；只重跑 base（如已有修改则保持最新）
2. 检查 `prompts/xinxin-ip-contract.md` 是否包含了该部分身份约束（头发、侧边饰品、身材占比）
3. 在 `prompts/rows/v2/*.md` 中为该状态补一句“identity lock hard constraint”重申（尽量不改动动作文案）
4. 单独重跑问题状态；再对比与其他状态首尾帧的一致性

### 重跑优先级
- P0：`idle`（这是可读基准，若漂移，先定基准）
- P1：`review` / `waiting`（最容易暴露微偏差）
- P2：`running-right` / `running-left` / `running`

---

## B. 帧数/槽位错位类

### 症状
- 帧数不足或过多
- 帧间重叠、裁切、跨槽位
- 某些槽位明显空白

### 修复顺序
1. 核对对应状态的布局引导图片（`references/layout-guides/*.png`）是否仍为正确尺寸
2. 临时在该 state prompt 最后追加：
   - `Output: Exactly N full-body frames, one full pose per slot, no empty slots.`
   - `Do not clip pose at slot boundaries.`
3. 重跑该 state 单独作业
4. 如仍异常，检查 `imagegen-jobs.json` 的 `prompt_file` 是否指向该 state 的 v2 prompt

### 重跑优先级
- P0：`idle` / `waiting`（对齐布局最敏感）
- P1：`running-right` / `running-left`（高帧数更易越界）
- P2：其余状态

---

## C. Loop 不闭合/抖动类

### 症状
- 播放首尾出现跳变
- 运动方向不稳定，节奏抖动

### 修复顺序
1. 在状态 prompt 内加入“first frame equals last frame semantics”
2. 增加 1-2 行节奏约束，例如：
   - `Keep loopable: first and last frame are pose-compatible.`
   - `No sudden amplitude spikes.`
3. 只重跑该状态，若是 `running-right` 同步检查 `running-left`
4. 通过后再看 `review` 与 `waiting` 的收敛感

---

## D. 风格禁区违反

### 症状
- 出现阴影、渐变、地面阴影、发光、模糊、粒子、符号
- 出现文本、标签、UI、边框、地景、漂浮物

### 修复顺序
1. 在对应 v2 prompt 末尾强化禁令（独立段落）
2. 增补一句：`Do not add any detached effects or visual effects except pose/outline/face changes.`
3. 单独重跑该 state
4. 若多个状态全部出现同类问题，先重跑 `idle` 做基底风格校准，再批量复跑其他

---

## E. 低可读性 / 模糊失真类

### 症状
- 小尺寸下动作细节不清
- 成员/肢体粘连，轮廓糊成一坨

### 修复顺序
1. 调整该状态 prompt，明确“chibi readability at 192x208”。
2. 明确要求“no anti-aliased high-detail edges”
3. 降低动作幅度，保证关键姿态点更突出（肩、膝、手）
4. 重跑该状态

---

## F. 批量失败应对（多状态同时不合格）
1. 清点是否有配置偏差：
   - `pet_request.json` 与 v2 版本字段
   - `ip_contract_file`、`ip_contract_version`
   - `xinxin-warm-lines.zh-CN.json` 版本
2. 检查全部状态是否都指向 `prompts/rows/v2/*.md`
3. 暂停 `running-left`，先重跑 `idle` + `running-right`，再镜像修复 `running-left`（如有必要）
4. 用顺序执行：
   - `idle -> waiting -> running -> review -> waving -> jumping -> running-right -> running-left -> failed`

---

## G. 复核通过后动作
1. 重新导出 `frames/*` 与 `final/*`（spritesheet）
2. 在 `qa/run-summary.json` 记录通过结果与当前提示版本
3. 保留 1 个回滚点（上一个稳定 `spritesheet`）以备回退

---

### 备注
- 强调最小改动原则：一次只改一个状态，避免连锁偏移。
- 默认禁止“全量重跑”，除非 base 或合同文本发生结构性变更。
